## 2. 新闻接口 — 公开访问需 @Anonymous

**问题描述：**
`TourismNewsController` 的 list 和 detail 端点有 `@PreAuthorize` 权限限制，前端公开页面（不带 token）访问会返回 401/403。

**后端需修复：**
```java
// TourismNewsController.java
@Anonymous  // 新增注解，绕过登录检查
@GetMapping("/list")
@PreAuthorize("@ss.hasPermi('tourism:news:list')")  // @Anonymous 下 @PreAuthorize 不生效

@Anonymous
@GetMapping("/{id}")
@PreAuthorize("@ss.hasPermi('tourism:news:query')")
```

**前端对应代码：**
```typescript
// src/api/news.ts
export const listNews = (query: NewsQuery) => {
  return request.get<any, any>('/tourism/news/list', { params: query })
}
export const getNews = (id: number) => {
  return request.get<any, any>(`/tourism/news/${id}`)
}
```

**状态：** 前端已就绪，后端加 `@Anonymous` 注解即可

# 待优化清单

> 本次 CEO Review 发现的问题，记录于此。涉及用户体验、技术债、运维可观测性等，非发布阻断但需后续处理。

## OPT-1. 报名审核结果通知机制

**问题描述：**
当前活动报名为申请-审核制，用户提交报名后无法得知审批结果（通过/拒绝）。用户旅程在提交报名后断裂——提交了，然后什么都没有了。

**影响：**
- 用户不知道自己的报名是否被审核
- 管理员审批通过/拒绝后，用户无法感知
- 用户会重复报名或质疑系统

**后端需实现：**
- 管理员审批操作后，发送站内消息通知用户
- 或接入邮件通知（`tourism_user` 表需有 email 字段）
- 消息内容：活动名称、审批结果（通过/拒绝）、审批意见

**前端需实现：**
- 用户中心"我的报名"页面，查看报名状态和审核结果
- 待后端确认通知机制后前端再接入

**优先级：** P1
**依赖：** 后端站内消息系统或邮件系统

## OPT-3. 后端日志 errorType 分类

**问题描述：**
`ActivityController.apply()` 捕获异常后统一打印通用日志，运维无法区分：
- 409 Conflict（用户重复报名）— 应归类为业务冲突
- 400 参数校验失败 — 应归类为客户端错误
- 500 服务器错误 — 应归类为系统错误

**后端需实现：**
```java
// 区分异常类型，日志中打印 errorType
if (e instanceof BusinessException) {
    log.warn("Activity application business error: {}, activityId={}, userId={}",
             ((BusinessException) e).getCode(), activityId, userId);
} else {
    log.error("Activity application system error: {}", e.getMessage(), e);
}
```

**影响：** 线上问题定位速度，运维友好度

**优先级：** P2
**工作量：** 小（约 1-2h）
