# AI开发工程规则

## 执行入口

当接收到开发任务时，必须先执行以下步骤：

1. 检查是否存在对应 task 文件  
2. 若不存在，必须先创建 task 文件  
3. 禁止跳过 task 直接进入编码  
4. 按开发流程逐步执行  

## 一、基础规则

1. 必须使用中文与用户交流  
2. 所有输出文档必须为 Markdown（.md）格式  
3. 严格按照规定目录创建文件，禁止随意创建目录  
4. 所有文件命名必须符合规范  


## 二、代码开发规范

### 1. 命名规范
- 变量、方法必须语义化，见名知意  
- 禁止拼音命名（常见缩写如 id、url 除外）  

### 2. 方法规范
- 单一职责：一个方法只做一件事  
- 方法长度 ≤ 50 行  
- 嵌套层级 ≤ 3 层，优先使用 early return  

### 3. 结构规范
- 相同逻辑必须抽取复用，禁止重复代码  
- 禁止保留无用代码（dead code）  

### 4. 可读性
- 禁止炫技写法  
- 优先保证代码清晰易读  


## 三、注释规范

1. 注释必须解释“为什么”，而不是“做什么”  
2. 所有 public 方法必须包含完整注释  

标准格式：

```

/**

* 功能说明
* 为什么这样实现（核心逻辑说明）
*
* @param 参数说明
* @return 返回值说明
  */

```

## 四、白盒测试规范

1. 所有业务代码必须编写对应白盒测试类  

2. 测试必须覆盖：
- 正常流程（success）  
- 边界条件（boundary）  
- 异常情况（exception）  

3. 命名规范：
```
test_方法名_场景

```

示例：
```

testCreateUser_success
testCreateUser_emptyName
testCreateUser_nullParam

```
4. 判定规则：
- 所有测试必须通过  
- 任一测试失败，视为功能未完成  


## 五、目录结构规范

项目目录必须如下：

```

/cquet-doc/开发日志/
/cquet-doc/task/

```

测试代码必须放在服务目录下：

```

src/test/java/

```


## 六、Task任务规范

文件路径：

```

/cquet-doc/task/任务名_YYYY-MM-DD.md

```

示例：

```

/cquet-doc/task/用户注册功能_2026-03-25.md

```

创建要求：

- 必须在开发前创建  
- 不允许开发完成后补写  

模板：

```

# 任务：XXX

## 1. 任务背景

## 2. 业务流程

## 3. 功能拆解

## 4. 技术方案

## 5. 风险点

```


## 七、开发日志规范

文件路径：

```

/doc/开发日志/功能名_YYYY-MM-DD.md

```

示例：

```

/doc/开发日志/用户注册功能_2026-03-25.md

```

生成要求：

- 必须在测试全部通过后生成  

模板：

```

# 功能名称 - YYYY-MM-DD

## 1. 功能描述

## 2. 实现内容

## 3. 测试情况

* 总用例：
* 通过：
* 失败：

## 4. 问题记录

## 5. 结论

## 八、开发流程

必须按以下顺序执行：

1. 创建 task 文档  
2. 编写业务代码  
3. 进行代码审阅（检查逻辑 / 边界 / 异常）  
4. 编写白盒测试  
5. 执行测试  
6. 修复所有失败测试  
7. 重复执行测试直至全部通过  
8. 生成开发日志  

## 九、错误处理

1. 测试失败时必须：
- 分析失败原因  
- 修复代码  
- 重新执行测试  

2. 所有问题必须记录到开发日志  

3. 不允许忽略失败测试或跳过修复  


## 十、任务完成条件

在任务结束前必须确认：

1. 已创建 task 文件  
2. 已完成代码开发  
3. 已编写白盒测试  
4. 所有测试已通过  
5. 已生成开发日志  

若任一未完成，不允许结束任务
