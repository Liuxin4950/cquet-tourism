# 一、前端问题清单

## A. P0 阻断项：先修，不然联调和演示会出问题

### FE-001 登录响应解析不一致

**现象**
 `auth store` 登录后按 `res.data.code / res.data.token / res.data.userInfo` 读取响应，但接口规范里登录成功返回更像是根层级 `code` 和 `token`，不是 `res.data.token`。

**风险**

- 登录明明成功，前端却拿不到 token
- `isAuthenticated()` 失效
- 登录后跳转和鉴权链路异常

**涉及文件**

- `src/stores/auth.ts`
- `src/api/auth.ts`

**建议修复**

- 统一登录响应解析逻辑，兼容两种结构：
  - `res.code / res.token`
  - `res.data?.code / res.data?.token`
- 登录成功后立刻调用一次 `fetchUserInfo()`
- 不再依赖登录接口直接返回完整 `userInfo`

**验收标准**

- 正常登录后 localStorage 中一定写入 token
- 刷新页面后仍保持登录
- `/profile` 可正常进入

------

### FE-002 获取用户信息字段映射错误

**现象**
 用户中心里手机号用的是 `userInfo?.phone`，但接口规范字段是 `phonenumber`。

**风险**

- 页面显示空手机号
- 用户信息展示不完整

**涉及文件**

- `src/views/ProfileView.vue`
- `src/stores/auth.ts`
- 类型定义相关位置

**建议修复**

- 建立统一 `UserProfile` 类型
- 在 store 中做字段标准化：
  - `phonenumber -> phone`
  - 或页面统一使用 `phonenumber`

**验收标准**

- 登录后用户昵称、手机号、头像都能正确显示

------

### FE-003 活动报名成功判断逻辑错误

**现象**
 `activity store` 的 `apply()` 用的是 `res.code`，而其他地方多为 `res.data.code` 或后端统一响应对象。

**风险**

- 后端报名成功，前端仍提示失败
- 用户重复点击产生脏数据或重复请求

**涉及文件**

- `src/stores/activity.ts`

**建议修复**

- 报名接口返回值统一解析
- 封装一个 `unwrapResponse()` 或统一响应适配器
- 对以下场景做分支：
  - 成功
  - 未登录 401
  - 重复报名 409
  - 参数错误 400
  - 网络失败

**验收标准**

- 报名成功时一定出现成功提示
- 重复报名有明确提示
- 未登录时引导去登录
- 网络异常时有 toast

------

### FE-004 新闻/公告公开接口依赖后端匿名开放

**现象**
 新闻和公告前端按公开页面设计，但后端若未加匿名访问，会直接 401/403。

**风险**

- 用户访问公开资讯页被强制跳转登录
- 公开站点逻辑被破坏

**涉及文件**

- `src/api/news.ts`
- `src/api/notice.ts`
- `src/api/request.ts`
- 对应后端控制器

**建议修复**
 前端：

- 对新闻/公告接口 401 不直接粗暴跳登录，至少允许显示“当前内容不可访问”
   后端：
- 按你们文档加 `@Anonymous`

**验收标准**

- 未登录用户可正常看新闻和公告列表、详情

------

### FE-005 构建环境不可复现

**现象**
 压缩包里带了 `node_modules`，当前构建链不稳定，`vite / vue-tsc / rollup` 相关执行存在环境问题。

**风险**

- 换机器跑不起来
- Codex 无法稳定实施
- 后续每次改动都难验证

**涉及范围**

- 项目根目录依赖管理
- `package.json`
- 仓库结构

**建议修复**

- 删除仓库中的 `node_modules`
- 提交标准的 `package-lock.json`
- 明确 Node 版本，例如 `Node 18/20`
- 增加 `.nvmrc`
- 让 Codex 先完成“干净安装可运行”

**验收标准**

- `npm install`
- `npm run dev`
- `npm run build`
   全部可在干净环境通过

------

## B. P1 高优先级：影响系统完整性

### FE-006 个人中心只是占位页，不是完整业务页

**现象**
 `ProfileView` 当前只是：

- 基本信息展示
- 退出登录
- 收藏/报名/留言三块空状态

**风险**

- 毕设答辩时会被认为“用户中心未完成”
- 用户闭环断裂

**涉及文件**

- `src/views/ProfileView.vue`

**建议修复**

- 重构为 tab 式用户中心：
  - 我的资料
  - 我的收藏
  - 我的报名
  - 我的评论

**验收标准**

- 用户中心至少具备真实的页面组织结构
- 即使接口未全接通，也有完整占位与交互框架

------

### FE-007 登录/注册页存在死链接

**现象**
 登录/注册页有：

- `/privacy-policy`
- `/terms`

但路由中没有对应页面。

**风险**

- 用户点击进入 404
- 答辩时显得“不完整”

**涉及文件**

- `src/views/LoginView.vue`
- `src/views/RegisterView.vue`
- `src/router/index.ts`

**建议修复**
 二选一：

- 先补两个静态页面
- 或临时去掉链接

**验收标准**

- 不存在可点击死链

------

### FE-008 导航缺少个人中心入口

**现象**
 当前登录后 navbar 只显示用户名和退出，没有明确进入 `/profile` 的入口。

**风险**

- 功能存在但用户找不到
- 用户路径不完整

**涉及文件**

- `src/components/layout/NavBar.vue`

**建议修复**

- 桌面端用户名可点击进入个人中心
- 移动端菜单加入“个人中心”
- 退出登录与个人中心分开

**验收标准**

- 登录后 1 次点击可进入用户中心

------

### FE-009 注销逻辑重复跳转

**现象**
 `authStore.logout()` 里会跳首页，组件中注销后又再次 `router.push('/')`。

**风险**

- 冗余导航
- 行为不清晰，后续维护容易出问题

**涉及文件**

- `src/stores/auth.ts`
- `src/components/layout/NavBar.vue`
- `src/views/ProfileView.vue`

**建议修复**

- 跳转职责统一放一处
- 推荐：store 只清状态，页面层决定跳转

**验收标准**

- 注销流程只有一次明确导航

------

### FE-010 认证状态仅依赖本地 token

**现象**
 当前 `isAuthenticated()` 本质上只是 `!!token`。

**风险**

- token 过期但页面仍显示已登录
- 直到请求 401 才发现失效

**涉及文件**

- `src/stores/auth.ts`
- `src/main.ts` 或 App 初始化位置

**建议修复**

- 应用启动时执行一次 `fetchUserInfo()`
- 拉取失败就清 token
- 增加 `isAuthReady` 状态避免闪烁

**验收标准**

- 过期 token 刷新页面后不会伪装成已登录

------

## C. P2 中优先级：质量和可维护性问题

### FE-011 API baseURL 使用不统一

**现象**

- `request.ts` 已有 `baseURL`
- `auth.ts` 又手动拼 `API_BASE`
- 其余旅游接口使用相对路径

**风险**

- 开发环境、生产环境行为不一致
- 容易出现代理/跨域问题
- 接口维护混乱

**建议修复**

- 所有 API 统一走 `request.ts`
- 移除各 API 文件重复拼接 baseURL 的写法

**验收标准**

- 全项目 API 地址策略一致

------

### FE-012 TypeScript 类型约束弱，`any` 偏多

**现象**
 store 和接口返回大量 `any`。

**风险**

- 字段错误无法编译发现
- 前后端契约问题隐藏太深

**建议修复**

- 先为这些实体补类型：
  - `ApiResponse<T>`
  - `LoginResponse`
  - `UserProfile`
  - `Activity`
  - `ActivityApplication`
  - `NewsItem`
  - `NoticeItem`

**验收标准**

- 核心 store/API 无裸 `any`
- 关键页面 props 和返回值有明确类型

------

### FE-013 `v-html` 渲染正文存在安全风险

**现象**
 新闻/公告详情页若直接渲染 HTML，会有潜在 XSS 风险。

**风险**

- 后台内容可插入危险脚本
- 安全性不足

**建议修复**

- 毕设阶段最少增加说明：
  - 后端输出已清洗
- 更稳妥是加入前端内容净化

**验收标准**

- 富文本渲染策略明确，不是“直接裸插”

------

### FE-014 缺少系统化测试

**现象**
 当前没有单测/E2E。

**风险**

- 每次改功能都靠人工点
- 联调时回归成本高

**建议修复**
 第一阶段至少补：

- auth store 单测
- activity store 单测
- login/register 表单验证测试

**验收标准**

- 核心链路至少有基础测试覆盖

------

## D. P3 体验层问题：不阻断，但影响答辩观感

### FE-015 视觉风格不统一

**现象**
 首页/内容页偏高端文旅，登录注册页偏 SaaS/插画风。

**风险**

- 品牌感不一致
- 项目像拼起来的两套站点

**建议修复**

- 登录/注册页色彩、字体、背景语言向主站靠齐
- 保留动效，但统一视觉体系

------

### FE-016 用户中心信息架构弱

**现象**
 三块内容直接纵向堆叠，缺乏 tab、筛选、层级感。

**建议修复**

- 改成左侧资料卡 + 右侧 tab 内容区
- 或顶部 tab + 下方统一内容容器

------

### FE-017 错误/空状态虽有基础，但缺少业务引导

**现象**
 空状态大多是“暂无数据”，但缺少 CTA。

**建议修复**

- 我的收藏空状态：去景区/场馆/活动
- 我的报名空状态：去活动列表
- 我的评论空状态：去最近活动
