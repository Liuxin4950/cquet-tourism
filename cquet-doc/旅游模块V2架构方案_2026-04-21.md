# 旅游模块 V2 架构方案

本文档用于重新定义重庆文旅项目的一期地基，目标是让系统更符合“文旅资源管理后台”的毕设主线，同时为后续二期扩展保留清晰边界。

## 1. 一期定位

一期系统定位为：

- 面向管理员的文旅资源管理后台
- 核心对象是景区、场馆、活动、资讯、公告
- 支持活动申报审核，但不引入“用户报名”业务链路
- 前台用户侧以浏览展示为主，互动能力不作为一期地基

结论：

- `用户报名` 不属于一期核心范围，不建议在一期数据库中建设报名表、报名审批表、报名状态流转表
- `收藏`、`评论` 不应成为一期核心架构的设计中心，可作为后置扩展能力按时间安排决定是否实现

## 2. 现状评估

结合当前代码、数据库和开题报告，一期地基存在以下问题：

### 2.1 领域边界不够清晰

当前旅游模块同时承载了：

- 资源管理
- 内容发布
- 活动审核
- 前台互动预留

这会让论文主线发散。对于毕业设计而言，更合理的口径应是：

- 一期先完成后台资源治理
- 二期再扩展用户互动与参与行为

### 2.2 主键类型不统一

当前代码中同时存在 `Long` 与 `Integer` 混用的情况，而数据库中多处外键已是 `bigint`。这会带来：

- 关联查询转换成本增加
- 代码可维护性下降
- 后续扩容和统一接口建模困难

V2 统一原则：

- 旅游模块所有主键、外键统一使用 `BIGINT`
- Java 实体统一使用 `Long`

### 2.3 图片模型仍然处于过渡态

当前模型中同时存在：

- 主表中的 `cover_image`
- 图片资源表
- 景区/场馆图片关联表

这说明“封面图”和“图片资源关系”还没有明确谁是唯一事实来源。

V2 统一原则：

- 文件统一进入 `tourism_media`
- 主表只保留 `cover_media_id`
- 多图通过关联表维护，不再直接在业务主表保存图片 URL

### 2.4 审核状态与审核历史混在一起

当前活动审批主要依赖 `tourism_activity_approval` 历史记录表，再由业务代码推导当前审核状态，容易导致：

- 列表查询性能不稳定
- 服务层出现 N+1 历史查询
- 当前状态与历史记录口径不一致

V2 统一原则：

- `tourism_activity.audit_status` 保存当前审核状态
- `tourism_activity_audit_log` 仅保存审核历史

### 2.5 前台互动能力不应反向塑造一期地基

当前库中存在：

- `tourism_user_collection`
- `tourism_activity_comment`

这些表并不适合作为一期核心架构的出发点。它们可以保留为二期规划，但不应影响一期主模型的稳定性。

## 3. 一期建议保留的核心业务

建议一期只保留下列业务主线：

### 3.1 景区管理

- 景区基础信息维护
- 景区图片维护
- 景区状态管理

### 3.2 场馆管理

- 场馆基础信息维护
- 场馆设施维护
- 场馆图片维护
- 场馆状态管理

### 3.3 活动管理

- 活动信息维护
- 活动所属场馆关联
- 活动审核流程
- 活动发布/停用状态控制

### 3.4 内容管理

- 新闻资讯发布
- 通知公告发布

## 4. 一期明确不纳入核心地基的能力

以下能力不建议纳入一期核心数据库设计：

- 用户报名活动
- 报名审批
- 用户消息中心
- 个性化推荐

其中“用户报名”应明确排除在一期之外；收藏、评论等能力即使后续保留，也不建议反向决定一期地基。

## 5. V2 数据模型设计原则

### 5.1 规范化原则

- 多值属性必须拆表
- 字典属性独立建表
- 关系通过关联表维护
- 当前状态与历史日志分离

### 5.2 命名原则

- 表名统一使用 `tourism_` 前缀
- 主实体、字典表、关联表语义清晰
- 避免 `image_url`、`devices` 这类承载多义的字段设计

### 5.3 与若依集成原则

- 继续复用若依的 `sys_user`、`sys_role`、`sys_menu`
- 旅游模块不自行复制用户体系
- 审计字段保持与若依风格一致：`create_by`、`create_time`、`update_by`、`update_time`、`remark`、`del_flag`

## 6. V2 一期推荐表结构

### 6.1 主表

- `tourism_scenic_spot`
- `tourism_venue`
- `tourism_activity`
- `tourism_news`
- `tourism_notice`

### 6.2 资源与字典表

- `tourism_media`
- `tourism_facility`

### 6.3 关联表

- `tourism_scenic_spot_media`
- `tourism_venue_media`
- `tourism_venue_facility`

### 6.4 审核日志表

- `tourism_activity_audit_log`

## 7. 关键建模说明

### 7.1 景区与场馆

景区和场馆是最核心的资源主实体，主表中只存稳定的单值属性：

- 名称
- 分类或等级
- 地址
- 区县
- 联系方式
- 坐标
- 封面资源
- 状态

多张图片通过关联表维护，不再在主表中直接存多图字段。

### 7.2 媒体资源

`tourism_media` 负责统一管理图片资源元数据：

- 文件名称
- 原始名称
- 文件 URL
- 大小
- MIME 类型
- 存储方式

这样可以让景区、场馆、新闻等都共享统一的资源管理模式，并与若依上传体系衔接。

### 7.3 活动

活动属于“依附于场馆的业务实体”，必须与场馆建立外键关联。

活动主表中建议直接保存：

- 当前审核状态 `audit_status`
- 发布状态 `status`

审核历史由 `tourism_activity_audit_log` 记录，这样列表查询不必再反复扫描审核历史推导当前状态。

### 7.4 设施

设施属于字典型基础数据，不应把设施名称直接写在场馆表中，也不应使用 JSON 或逗号拼接。

正确方式：

- `tourism_facility` 保存设施字典
- `tourism_venue_facility` 保存场馆与设施关系

## 8. 一期与二期分层建议

### 8.1 一期

- 管理后台
- 景区管理
- 场馆管理
- 设施管理
- 图片资源管理
- 活动管理
- 活动审核
- 新闻与公告管理

### 8.2 二期

- 用户报名
- 报名审批
- 用户消息中心
- AI 推荐与个性化服务

说明：

- 收藏、评论可根据论文篇幅与实现时间决定是否放入二期
- 但它们不建议进入一期基础库设计

## 9. 对当前旧表的处置建议

### 9.1 一期建议删除或暂不纳入 V2 的表

- `tourism_user_collection`
- `tourism_activity_comment`

说明：

- 不是说这些能力永远不要
- 而是它们不应该进入一期基础版建模

### 9.2 一期明确不新增的表

- `tourism_activity_registration`
- `tourism_activity_signup`
- `tourism_activity_application_user`

只要这些表对应的是“用户报名活动”这条业务线，一期都不建议落库。

## 10. 论文可采用的表述

可以在论文中采用如下口径：

> 本系统一期以文旅资源管理后台为核心，围绕景区、场馆、活动、资讯与公告构建统一的数据模型。对于用户报名、收藏、评论等前台互动功能，考虑到项目阶段目标与系统复杂度控制，将其划分为二期扩展能力，不纳入一期核心数据库设计。数据库设计遵循第三范式，将图片资源、设施字典、多对多关系以及审核日志进行拆分建模，从而提升系统的可维护性、扩展性与数据一致性。

## 11. 推荐改造顺序

1. 统一所有旅游模块主外键为 `BIGINT / Long`
2. 用 `tourism_media + 关联表` 替换主表图片 URL 真值
3. 为活动主表补齐 `audit_status`
4. 将审核历史收口到 `tourism_activity_audit_log`
5. 将收藏、评论、报名等能力移出一期核心 schema
