# 任务：代码补强

## 1. 任务背景

毕业设计代码已基本完成，但需要工程化完善以通过答辩评审。代码审查发现以下问题：
- Service 层业务校验失败时返回 `false`，前端无法获得明确错误信息
- 活动冲突检测缺少边界测试用例
- 单元测试已存在（质量较好），但部分失败场景未覆盖

## 2. 业务流程

### 活动冲突检测流程
```
用户提交活动 → 名称唯一性检查 → 同场馆时间段冲突检查 → 写入数据库
                                         ↓（发现冲突）
                               抛出 ServiceException
```

**冲突判定规则：**
- 冲突：`existing.startTime < new.endTime AND existing.endTime > new.startTime`
- 不冲突：`existing.endTime <= new.startTime OR existing.startTime >= new.endTime`

### 边界用例
| 场景 | 活动A | 活动B | 预期结果 |
|------|-------|-------|----------|
| 相邻 | 09:00-10:00 | 10:00-11:00 | 不冲突 ✓ |
| 部分重叠 | 09:00-11:00 | 10:00-12:00 | 冲突 |
| 完全包含 | 09:00-12:00 | 10:00-11:00 | 冲突 |
| 不同场馆 | 10:00-12:00(场馆1) | 10:00-12:00(场馆2) | 不冲突 |

## 3. 功能拆解

1. **补单元测试** — 4个时间冲突边界测试用例（相邻/不同场馆/部分重叠/完全包含）
2. **补充异常处理** — TourismActivityServiceImpl.create() 和 TourismScenicSpotServiceImpl.addScenicSpot() 抛出明确 ServiceException
3. **更新测试** — 相应测试用例从 assertFalse 改为 assertThrows(ServiceException)

## 4. 技术方案

### 异常处理改进
**Before：** `return false; log.error("该景区已存在");`
**After：** `throw new ServiceException("景区名称已存在");`

**好处：**
- 前端可获得明确错误信息（通过 GlobalExceptionHandler 处理）
- Spring 事务自动回滚（异常自动触发 rollback）
- 与若依框架标准模式一致

### 测试改进
**Before：** `assertFalse(result)` — 知道失败但不知原因
**After：** `assertThrows(ServiceException.class, ...); assertEquals("具体消息", e.getMessage())` — 精确定位

## 5. 风险点

- 异常处理改动涉及 Service → Controller 契约变化，但 Controller 已有 try-catch 兼容，新异常可被 GlobalExceptionHandler 正确处理
- 单元测试覆盖已经较完整（13+19=32个测试），风险低
