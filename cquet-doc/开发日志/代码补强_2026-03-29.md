## 1. 功能描述

## 2. 实现内容

### 2.1 异常处理机制改进

**问题：** Service 层业务校验失败时仅返回 `false`，前端只能收到模糊错误提示。

**改进：** 在业务校验不通过时抛出 `ServiceException`（若依框架标准业务异常），携带明确错误消息。

**涉及方法：**
- `TourismActivityServiceImpl.create()` — 名称重复、时段冲突时抛出明确异常
  - `"活动名称已存在"` — 名称重复
  - `"该场馆在所选时间段已有其他活动安排"` — 时段冲突
- `TourismScenicSpotServiceImpl.addScenicSpot()` — 名称重复时抛出异常
  - `"景区名称已存在"`

**技术细节：**
- `ServiceException` 由 `GlobalExceptionHandler` 统一处理，返回 `Result.error(消息)`
- `ServiceException` 继承 `RuntimeException`，触发 Spring `@Transactional` 自动回滚

### 2.2 单元测试边界用例补充

**问题：** 活动时间冲突检测只有"有冲突时拒绝"一个用例，缺少边界条件测试。

**改进：** 新增 4 个边界测试用例（位于 `TimeConflictBoundaryTests` 嵌套类）：

| 测试用例 | 场景 | 预期 |
|----------|------|------|
| `create_Success_WhenTimeSlotsAreAdjacent` | 09:00-10:00 vs 10:00-11:00 | 不冲突 ✓ |
| `create_Success_WhenDifferentVenue` | 同时间段，venueId=1 vs venueId=2 | 不冲突 ✓ |
| `create_Fails_WhenPartiallyOverlapping` | 09:00-11:00 vs 10:00-12:00 | 冲突 |
| `create_Fails_WhenContained` | 09:00-12:00 vs 10:00-11:00 | 冲突 |

**冲突判定逻辑（已验证正确）：**
```java
qw.lt(TourismActivity::getStartTime, activity.getEndTime())   // existing.start < new.end
   .gt(TourismActivity::getEndTime, activity.getStartTime()); // existing.end > new.start
```

### 2.3 测试断言更新

原有测试断言 `assertFalse(result)` 对应"返回false"场景，改为 `assertThrows(ServiceException.class, ...)` 配合 `assertEquals(具体消息)`，实现精确断言。

## 4. 问题记录

**问题1：** `create_Fails_WhenPartiallyOverlapping` 和 `create_Fails_WhenContained` 初版测试用 `assertFalse(result)` 断言，但 Service 层已改为抛出异常，导致测试变成 ERROR 而非 FAILURE。

**解决：** 将断言改为 `assertThrows(ServiceException.class, ...)` 并增加 `assertEquals(消息)` 精确验证异常内容。

**问题2：** 测试用例中时间边界理解有误。

**分析：** `existing.endTime <= new.startTime` 时不冲突（如 09:00-10:00 与 10:00-11:00）。当前实现用 `.lt()` 和 `.gt()`（严格小于/大于），完全符合预期边界行为。
