## 一、项目概述

### 1.1 项目背景

基于若依（RuoYi）框架二次开发的重庆文旅管理系统，用于管理重庆 A 级景区、文化场馆和特色活动，为本地居民和游客提供一站式文旅服务平台。

**技术栈：**
- 后端：Spring Boot 3.3.2 + MyBatis-Plus 3.5.5 + Spring Security + JWT
- 前端（管理后台）：Vue 2 + Element UI + echarts
- 前端（用户端）：Vue 3 + Vite + TailwindCSS + MapLibre GL
- 数据库：MySQL + Druid 连接池

### 1.2 模块架构

```
cquet-demo/
├── cquet-admin/          # 主应用入口
├── cquet-framework/      # 框架扩展（安全、认证）
├── cquet-system/         # 系统管理（用户/角色/菜单）— 若依基础模块
```

### 2.2 多表关联事务（最硬核亮点）

**场景：** 管理员上传景区时，需要同时维护景区主表和图片关联表。

**实现：** 使用 `@Transactional(rollbackFor = Exception.class)` 保证原子性：

```java
public boolean addScenicSpot(TourismScenicSpot scenicSpot) {
    // 1. 插入景区主表
    tourismScenicSpotMapper.insert(scenicSpot);
    // 2. 批量插入图片关联记录
    if (scenicSpot.getImageIds() != null && !scenicSpot.getImageIds().isEmpty()) {
        for (Integer imageId : scenicSpot.getImageIds()) {
            TourismScenicSpotImage rel = new TourismScenicSpotImage();
            rel.setScenicSpotId(scenicSpot.getId().intValue());
            rel.setImageId(imageId);
            scenicSpotImageMapper.insert(rel);
        }
    }
    return true;
}
```

**答辩话术：** "系统使用 Spring 的声明式事务管理，确保了景区主表和图片关联表的数据一致性。如果关联记录插入失败（如外键约束），主表操作会自动回滚。"

### 2.3 活动时间冲突检测（最有技术含量）

**冲突判定规则：**
- 冲突：`existing.startTime < new.endTime AND existing.endTime > new.startTime`
- 不冲突：相邻时间段（endTime = startTime）视为连续活动，允许

**实现代码：**
```java
qw.eq(TourismActivity::getDelFlag, "0")
  .eq(TourismActivity::getVenueId, activity.getVenueId())
  .lt(TourismActivity::getStartTime, activity.getEndTime())   // 已有.startTime < 新.endTime
  .gt(TourismActivity::getEndTime, activity.getStartTime());  // 已有.endTime > 新.startTime
```

**边界测试用例：**

| 用例 | 场景 | 预期 |
|------|------|------|
| 相邻不冲突 | 09:00-10:00 vs 10:00-11:00 | ✓ 允许 |
| 部分重叠 | 09:00-11:00 vs 10:00-12:00 | ✗ 拒绝 |
| 完全包含 | 09:00-12:00 vs 10:00-11:00 | ✗ 拒绝 |
| 不同场馆 | 同时间段 venueId=1 vs venueId=2 | ✓ 允许 |

### 2.4 数据大屏（最可视化）

`TourismDashboardServiceImpl` 提供 5 个统计维度：

1. **综合统计** — 景区/场馆/活动总数 + 待审核数量
2. **区县分布** — 各区县景区数量柱状图
3. **景区热度排行** — TOP 10 热门景区（按访问量）
4. **月度活动趋势** — 近6个月活动场次折线图
5. **地图分布** — 景区经纬度散点图

### 2.5 审批工作流

活动发布需经过审批：

```
发布申请 → PENDING → APPROVED（通过）
                       ↘ REJECTED（拒绝）
```

- 审批历史全程记录（审批人、时间、理由）
- 同一活动支持多次审批记录

## 三、工程化补强（本轮工作）

### 3.1 异常处理规范化

**改进前：**
```java
if (exists) {
    log.error("该景区已存在");
    return false;  // 前端无法获得明确错误消息
}
```

**改进后：**
```java
if (exists) {
    throw new ServiceException("景区名称已存在");  // GlobalExceptionHandler 统一处理
}
```

**改进范围：**
- `TourismActivityServiceImpl.create()` — 名称重复、时段冲突
- `TourismScenicSpotServiceImpl.addScenicSpot()` — 名称重复

**收益：**
- 前端可获得明确中文错误提示
- 异常自动触发 Spring 事务回滚
- 与若依框架标准模式一致

### 3.2 单元测试边界覆盖

**新增 4 个边界测试用例（`TimeConflictBoundaryTests`）：**
- `create_Success_WhenTimeSlotsAreAdjacent` — 相邻时间不冲突
- `create_Success_WhenDifferentVenue` — 不同场馆不冲突
- `create_Fails_WhenPartiallyOverlapping` — 部分重叠时拒绝
- `create_Fails_WhenContained` — 完全包含时拒绝

**修复预先存在的测试错误：**
- `TourismVenueServiceTest.removeVenueByIds_Success` — mock 期望与实际代码不符（代码用 update，测试写的是 delete）
