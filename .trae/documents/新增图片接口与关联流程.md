## 目标
- 新增图片记录接口，保存图片 URL 并返回生成的 `id`
- 前台在新增场馆或景区前先创建图片，拿到 `id` 后通过已有 `imageIds` 字段完成关联
- 保持与现有架构一致（Controller + Service + Mapper），沿用统一的响应 `Result`

## 接口设计
- POST `tourism/images`
  - 请求体：`{ "url": "https://..." }`
  - 返回：`{ id, url }`
  - 权限：`@PreAuthorize("@ss.hasPermi('tourism:image:add')")`
  - 日志：`@Log(title="图片管理", businessType=INSERT)`
- POST `tourism/images/batch`
  - 请求体：`["https://...", "https://..."]`
  - 返回：`[{ id, url }, ...]`
  - 权限：`@PreAuthorize("@ss.hasPermi('tourism:image:add')")`
- GET `tourism/images/{id}`
  - 返回：单条图片记录
  - 权限：`@PreAuthorize("@ss.hasPermi('tourism:image:query')")`
- DELETE `tourism/images/{ids}`
  - 批量删除图片记录（不影响已建立的关联表数据，前端谨慎使用）
  - 权限：`@PreAuthorize("@ss.hasPermi('tourism:image:remove')")`

## 数据模型与流程
- 使用现有表与实体：`TourismImage(id, url)`、`TourismScenicSpotImage`、`TourismVenueImage`
- 新增图片：插入一条 `tourism_image`，返回 `id`
- 新增场馆/景区：前端将图片 `id` 放入 `imageIds`（已存在，`TourismVenue:100`、`TourismScenicSpot:80`），服务层会自动写入关联表
- 详情查询：现有实现已读取关联并返回 `images` 列表（`TourismVenueServiceImpl:141-143`、`TourismScenicSpotServiceImpl:173-176`）

## 实现步骤
1. 新增 `TourismImageService` 与 `impl/TourismImageServiceImpl`
   - 方法：`create(TourismImage img)`, `createBatch(List<String> urls)`, `getById(Integer id)`, `removeByIds(List<Integer> ids)`
   - 直接使用 `TourismImageMapper`
2. 新增 `TourismImageController`
   - 路径前缀：`@RequestMapping("tourism/images")`
   - 以上四个接口方法，使用 `Result.success()/warn()` 封装
   - 参数校验：`url` 非空、长度限制；重复 URL 可插入或可选去重（默认不限制）
3. 权限点约定
   - `tourism:image:list`、`tourism:image:query`、`tourism:image:add`、`tourism:image:remove`
   - 与现有控制器一致添加 `@PreAuthorize` 与 `@Log`

## 前端使用方式
- 先调用 POST `tourism/images` 创建图片，拿到 `{ id }`
- 新增场馆或景区时，在请求体的 `imageIds` 中传入这些 `id`
- 详情页与列表无需改动，已有逻辑会返回 `images` 数组

## 兼容性与影响
- 不改动既有新增/修改场馆、景区的接口与流程，仅提供更便捷的图片创建入口
- 删除图片不影响已建立关联表；若需严格约束，可后续加入“禁止删除已关联图片”的校验

## 后续扩展（可选）
